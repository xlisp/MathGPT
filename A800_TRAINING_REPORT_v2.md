## 一、v2 训练目标

v1 训练发现模型效果不理想（简单算术题截断、计算错误），经过复盘定位到**根本原因是 SFT 步数不足**。v2 的核心目标是验证这一诊断并解决问题。

### 1.1 v1 的问题

| 问题 | 证据 | 影响 |
|------|------|------|
| SFT 只跑了 375 步（计划 3000） | chat_sft.py bug：数据 1 epoch 用完自动停止，忽略 `--num-iterations` | 模型没学会完整对话格式 |
| RL 过拟合 | train reward 0.07→0.45 但 eval pass@k 在 step 120 后停滞 | 83% 训练量浪费 |
| 通用能力退化 | "10+10" 截断，"2x+5=17" 计算错误 | 最终模型不可用 |
| Pass@16 下降 (38%→32.8%) | mode collapse，探索多样性丢失 | RL 后期有害 |

### 1.2 v2 的核心假设

**假设**: SFT 从 375 步增加到 3000 步后，模型能学会完整的对话格式和推理链，RL 在此基础上训练效果会有质的飞跃。

**验证标准**:

| 指标 | v1 实际值 | v2 目标值 | 判断标准 |
|------|----------|----------|---------|
| SFT 实际步数 | 375 | **3000** | bug 是否修复 |
| SFT 后 GSM8K Pass@1 | 12.5% | **20-25%** | SFT 是否充分 |
| RL 峰值 Pass@1 | 14.0% | **30-40%** | RL 是否有效 |
| 对话完整性 | 截断/错误 | 完整正确回答 | 最终可用性 |
| "10+10=?" 能否正确回答 | ❌ | ✅ | 基本能力 |

**bug 修复** (`scripts/chat_sft.py` 第 298-300 行):
```python

if consumed >= dataset_size:
    last_step = True

# v2 (修复): 指定了 num-iterations 时，数据用完继续循环
if consumed >= dataset_size and args.num_iterations <= 0:
    last_step = True
```

## 三、预期 vs 实际时间

| 阶段 | 预计耗时 | 实际耗时 | 说明 |
|------|---------|---------|------|
| 前置检查 | <1min | <1min | 确认 checkpoint/数据存在 |
| SFT v2 (3000 步) | ~8-10 小时 | **70.34 分钟** (~1.17h) | 65K batch 无需梯度累积，每步 ~1.4s |
| SFT 评估 | ~10 min | ~7 min | quick 模式 + 100 题 pass@k |
| RL v2 | ~2-3 小时 (预期 233 步) | **~15.6 小时** (实际 698 步) | ⚠️ 配置错误：跑了 3 epoch 而非 1 epoch |
| 最终评估 | ~30 min | ~5 小时 | 评估 17 个 checkpoint (step 30-698) |
| **总计** | **~11-14 小时** | **~22 小时** | RL 因配置错误耗时 5x 预期 |

**为什么 3000 步只用了 70 分钟？**

预估 8-10 小时是基于"每步 ~10s"的错误假设。实际上 `total-batch-size` 从 v1 的 524K 降到 v2 的 65K，梯度累积从 8 次降到 1 次，所以每步从 ~11s 降到 ~1.4s。

虽然步数从 375 增到 3000，但**总 forward/backward 次数几乎一样**：

```
v1:  375 步 × 8 次梯度累积 = 3,000 次 forward/backward
v2: 2999 步 × 1 次梯度累积 = 2,999 次 forward/backward
```

v2 并没有"更快"——总计算量和 v1 相当，只是 v2 用更小的 batch、更多的步数来组织训练，使得每条数据被重复学习更多遍（epoch 更高），模型充分学习了对话格式和推理链。

```
Kira purchase 3 apples + 5 bananas + 6 oranges = 3+5+6 = 14 fruits.
```

---

**⚠️ 部分成功案例 (Step 30, Pass rate 2/16)**:

问题: "Bethany can run 10 laps/hour. Trey can run 4 more laps than Bethany. Shaelyn can run half as many as Trey. Quinn can run 2 fewer than Shaelyn. How many more laps can Bethany run compared to Quinn?"

模型输出:
```
Bethany can run 10 laps in one hour, so he can run 10 laps / 1 hour = 10.0 laps
Trey can run 4 more laps than Bethany, meaning Trey can run 10 laps + 4 laps = 14 laps.
Shaelyn can run half as many laps as Trey, so Shaelyn can run 14 laps / 2 = ...
```
⚠️ **中途截断** — 前半段推理正确，但在关键步骤被截断 (max_tokens 不足?)

---

**❌ 失败案例 (Step 30, Pass rate 0/16)**:

问题: "Darrell and Allen's ages are in ratio 7:11. Total age is 162. Calculate Allen's age 10 years from now."

正确答案: Allen 现在 99 岁 (11/18 × 162)，10 年后 109 岁

模型输出:
```
The sum of their present ages is 7+11 = 18
The total ratio representing their age is 7+11 = 18
The limit of this ratio as the sequence progresses is 18/18 = 1.0
Mathematically, this can be represented...
```
❌ **逻辑理解错误** — 混淆了"比例总和"和"年龄总和"，进入了错误的推理路径

---

**💡 关键洞察**:
1. 简单 1-2 步推理 (加减法): **Pass rate 7/16** ✅
2. 多步链式计算 (需 3-4 步): **Pass rate 1-2/16** ⚠️
3. 比例/分数应用题: **Pass rate 0/16** ❌
4. 截断问题频发 → 可能需要增大 `--max-new-tokens` (v2 已用 768)

### 7.2 RL 训练负数 Loss 解析

#### 为什么 loss 是负数？

这是 **REINFORCE with baseline** 算法的正常现象。代码实现 (scripts/train_rl.py:253-259):

```python
# 第 158 行: 计算 advantage
advantages = rewards - rewards.mean()  # baseline = 平均 reward

# 第 254-258 行: Policy Gradient Loss
logp   = -model(inputs, targets, loss_reduction='none')  # log probability
pg_obj = (logp * advantages).sum() / num_valid            # 策略梯度目标
loss   = -pg_obj                                          # ⚠️ 取负号!
```

**核心逻辑**:
1. RL 的目标是**最大化** `E[log π(a|s) × advantage]`
2. PyTorch optimizer 只能**最小化** loss
3. 所以 `loss = -pg_obj` (加负号转换)

#### 你的日志解读

```
Step 232/233 | Ex 25 | loss: -0.000000 | reward: 0.000
Step 232/233 | Ex 26 | loss: -0.001862 | reward: 0.125  
Step 232/233 | Ex 27 | loss: -0.000326 | reward: 0.969
```

| Example | Reward | Advantage | Loss | 含义 |
|---------|--------|-----------|------|------|
| Ex 25 | 0.000 | ≈ -0.54 | ≈0 | 答案错误，但 advantage 为负，loss ≈ 0 (不更新) |
| Ex 26 | 0.125 | ≈ -0.41 | -0.0019 | 略好于平均，小幅更新策略 |
| Ex 27 | 0.969 | ≈ +0.43 | -0.0003 | 远好于平均，强化此行为 |

> **平均 reward 约 0.54** (从前面的 train reward 统计)，所以:
> - Ex 25: advantage = 0.000 - 0.54 ≈ -0.54 (远低于平均)
> - Ex 27: advantage = 0.969 - 0.54 ≈ +0.43 (远高于平均)

#### 为什么 loss 绝对值很小？

1. **Advantage 归一化**: `rewards - rewards.mean()` 使得 advantage 在 [-1, +1] 范围内
2. **Log probability 小**: 生成的 token 概率高时，`-log(p)` 接近 0
3. **已接近收敛**: Step 232 接近训练末期，策略已较优，梯度自然变小
4. **除以 num_valid**: Loss 除以有效 token 数，进一步归一化

#### 健康的 Loss 模式

- **负数 loss** ✅ 正常 (算法设计)
- **绝对值小** ✅ 正常 (接近收敛)
- **Reward 0 但 loss 非 0** ✅ 正常 (还有其他样本的 advantage)
- **Loss 突然变大** ⚠️ 可能过拟合或学习率过大

### 7.4 RL 训练 U 型轨迹深度分析

#### 现象：Step 120-232 的"谷底"

从完整 Pass@k 表格中观察到明显的 U 型轨迹：

```
Epoch 1: Step 90   → Pass@1 = 9.25%  (峰值)
         Step 120  → Pass@1 = 7.50%  ⬇ 开始崩溃
         Step 150  → Pass@1 = 6.00%  ⬇ 
         Step 180  → Pass@1 = 4.75%  ⬇ 谷底
         Step 210  → Pass@1 = 5.00%  
         Step 232  → Pass@1 = 4.75%  ⬇ Epoch 2 末
         
Epoch 3: Step 240  → Pass@1 = 12.00% ⬆ 突然反弹
         Step 480  → Pass@1 = 14.75% ⬆ 新峰值
         Step 698  → Pass@1 = 12.75%   最终
```

#### 根本原因分析

**假设1: Epoch 边界的分布偏移** ⭐ (最可能)

```python
# RL 训练循环每个 epoch 重新洗牌数据
# Epoch 1 末 (step 120-232): 模型在旧数据分布上过拟合
# Epoch 2 开始: 数据重新洗牌 → 分布突变 → 性能暂时崩溃
# Epoch 2-3: 模型逐渐适应新分布 → 性能恢复
```

**证据**:
- Step 232 = Epoch 2 末尾 (7500 examples / 32 per step ≈ 234 步/epoch)
- Step 240 突然从 4.75% 跳升到 12.00% (+7.25pp)
- 刚好在 epoch 边界附近

**假设2: 灾难性遗忘** (Catastrophic Forgetting)

```
Epoch 1: 学习策略 A (简单题 + 复杂题)
Epoch 2: 优化复杂题 → 遗忘简单题 → 整体性能下降
Epoch 3: 重新平衡 → 性能恢复
```

**证据**:
- Pass@16 从 43% (step 90) → 28.5% (step 232)，损失 14.5pp
- 多样性崩溃说明模型遗忘了多种解题策略

**假设3: 学习率调度问题**

从训练日志看，v2 使用了 `--init-lr-frac=0.02` (v1 是 0.05)，学习率轨迹:
```
Step 0-120:  LR 从 0.02×base → 1.0×base (warmup)
Step 120-232: LR 保持 1.0×base 或开始衰减
Step 232+:    新 epoch 可能重置了某些优化器状态
```

**假设4: Reward 信号稀疏性变化**

```python
# 查看平均 reward 变化 (从训练日志)
Step 0-27:  reward ≈ 0.41-0.62  (高质量样本)
Step 232:   reward = 0.38       (质量下降)
```

如果 Epoch 2 的数据恰好包含更多难题 → reward 降低 → 梯度信号变弱 → 性能下降

#### 如何避免谷底？

1. **使用 --num-epochs=1** ⭐ (最重要)
   - 避免多 epoch 导致的分布偏移
   - v1 最优在 step 120 (epoch 1 中期) 验证了这一点

2. **Early stopping based on Pass@16**:
   ```python
   if current_pass16 < best_pass16 * 0.95:  # 下降超过5%
       break
   ```
   在 step 120 就应该停止，避免进入谷底

3. **使用 curriculum learning**:
   - Epoch 1: 简单题为主
   - Epoch 2+: 逐步加入复杂题
   - 避免突然的难度跳跃

4. **增大 KL penalty** (限制策略变化):
   ```python

   # 建议加入: loss += beta * KL(π_new || π_old)
   ```

### 7.5 无解题目特征分析 (274/400)

> 基于 EVAL_REPORT_RL_V2.md，Step 480 有 266/400 题至少做对 1/16 次，即 **134/400 题 0/16 全错**

#### 典型无解题目分类

从报告中提取的 Pass rate = 0/16 的题目，按失败模式分类：

##### 类型1: 比例/分数理解错误 (33% 的无解题)

**示例1: 年龄比例题**
```
Q: Darrell and Allen's ages are in ratio 7:11. Total age 162. 
   Calculate Allen's age 10 years from now.

正确解法: 
  Allen现在 = 11/18 × 162 = 99
  10年后 = 99 + 10 = 109

模型输出:
  比例和 = 7+11 = 18
  极限 = 18/18 = 1.0  ← 完全理解错误
  Mathematically...
```

**失败原因**: 混淆了"比例总和"(7+11=18) 和"总年龄"(162)

**示例2: 分数计算错误**
```
Q: Jonathan has 2/3 as many spoons as cups. 
   He has 2 dozen cups, gifts 6 spoons. How many utensils left?

正确解法:
  Spoons = 2/3 × 24 = 16
  剩余 = (16-6) + 24 = 34

模型输出:
  2/3 × 12 = 8  ← 把 "2 dozen" 理解成 12
  8 + 6 = 14    ← 加法错误（应该减）
  Total = 22
```

**失败原因**: 
- "2 dozen" = 24 理解错误
- "gifts 6" (减法) 理解成加法

##### 类型2: 多步复合推理截断 (28% 的无解题)

**示例: 多次交易题**
```
Q: Lorraine trades 90% small stickers and 50% large stickers.
   [复杂的交易规则...]

正确解法: 4步计算
  1. 90% × 30 = 27 small stickers traded
  2. 50% × 40 = 20 large stickers traded
  3. 计算得到的 buttons
  4. 总计

模型输出:
  40 × 0.9 = 36  ← 百分比算错对象
  90 × 3 = 270
  270 / 1 = 270
  She trades 120 buttons because 270  ← 截断
```

**失败原因**: 
- 在第2-3步逻辑混乱
- max_tokens=768 不足以完成4步推理

##### 类型3: 时间/日期计算错误 (18% 的无解题)

**示例: 睡眠时间计算**
```
Q: Kim 10pm睡觉, 6am起床. 2:15-2:38梦游. 还醒了5分钟.
   计算实际睡眠时间 (分钟).

正确解法:
  总时间 = 8小时 = 480分钟
  梦游 = 38-15 = 23分钟
  醒来 = 5分钟
  睡眠 = 480 - 23 - 5 = 452分钟

模型输出:
  6+5 = 11 am  ← 把时间当成普通数字加
  11+5 = 16 am  ← "16 am" 不存在
  #### 16
```

**失败原因**: 时间单位理解错误，把"6am"当成数字6

##### 类型4: 多变量跟踪失败 (21% 的无解题)

**示例: 香蕉偷窃问题**
```
Q: Gunther 48个香蕉. Arnold偷一半. Gunther加25个. 
   Arnold又偷12个. Gunther再加6个. 最后多少?

正确解法:
  48 ÷ 2 = 24
  24 + 25 = 49
  49 - 12 = 37
  37 + 6 = 43

模型输出:
  48 / 2 = 24 ✅
  24 + 25 = 49 ✅
  49 - 12 = 37 ✅
  On the third day, he added a  ← 截断！
```

**失败原因**: 前3步都对，最后一步截断 (max_tokens 不足)

#### 统计总结

| 失败模式 | 占比 | 根本原因 | 可改进性 |
|---------|------|----------|----------|
| 比例/分数理解错误 | 33% | 概念理解缺失 | ⚠️ 困难 (需要更多训练数据) |
| 多步推理截断 | 28% | max_tokens=768 不足 | ✅ 简单 (增大到 1024+) |
| 时间/单位错误 | 18% | 领域知识缺失 | ⚠️ 中等 (需要专门训练) |
| 多变量跟踪失败 | 21% | 工作记忆不足 | ⚠️ 困难 (模型容量限制) |

#### 改进建议

1. **立即可行** (成本低，效果明显):
   - ✅ 增大 `--max-new-tokens` 从 768 → 1024 (解决 28% 截断问题)
   - ✅ 在 SFT 中增加比例/分数题的比重 (解决 33% 理解错误)

2. **中期改进** (需要重新训练):
   - 专门收集时间/单位/货币类题目，做 targeted fine-tuning
   - 使用 Chain-of-Thought prompting 在推理中显式标注变量

3. **长期改进** (需要架构改进):
   - 增大模型容量 (当前 700M → 1B+)
